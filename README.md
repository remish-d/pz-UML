# Practical lesson pz-UML
Побудова повердінкових UML-діаграм для проєктування інформаційних систем

## Тема - Система віддаленого управління вузлами комп'ютерної  мережі при реагуванні на кіберінциденти

## 1. Діаграма варіантів використання (Use Case Diagram)

Ця діаграма описує функціональність системи з точки зору користувачів (Адміністратора безпеки) та зовнішніх систем (Системи моніторингу/SIEM).

`code`

       useCaseDiagram
        actor "Адміністратор безпеки" as Admin
        actor "Система моніторингу (SIEM)" as SIEM
    
    package "Система управління вузлами" {
        usecase "Авторизація в системі" as UC1
        usecase "Перегляд сповіщень про інциденти" as UC2
        usecase "Ізоляція вузла від мережі" as UC3
        usecase "Збір логів та артефактів" as UC4
        usecase "Запуск скриптів очищення" as UC5
        usecase "Відновлення доступу вузла" as UC6
    }

    Admin --> UC1
    Admin --> UC2
    Admin --> UC3
    Admin --> UC4
    Admin --> UC5
    Admin --> UC6

    SIEM --> UC2
    UC3 <.. UC5 : <<extend>>

`code`

## 2. Діаграма послідовності (Sequence Diagram)

Відображає процес взаємодії між компонентами системи під час виконання конкретного сценарію: 
* **Ізоляція вузла після отримання сигналу про атаку.**

`code`

    sequenceDiagram
    autonumber
    participant A as Адміністратор
    participant UI as Веб-інтерфейс
    participant C as Контролер системи
    participant N as Мережевий вузол (Agent)

    A->>UI: Перегляд деталей інциденту
    UI->>C: Запит статусу вузла
    C-->>UI: Вузол "В мережі", виявлено аномалію
    
    A->>UI: Команда "Ізолювати вузол"
    UI->>C: Передача команди ізоляції
    
    C->>N: Блокування мережевих інтерфейсів (крім управлінського)
    Note over N: Виконання правил Firewall
    N-->>C: Підтвердження ізоляції
    
    C-->>UI: Оновлення статусу: "Ізольовано"
    UI-->>A: Візуальне підтвердження успішної операції
    
`code`

## 3. Діаграма діяльності (Activity Diagram)

Описує алгоритм роботи системи при обробці кіберінциденту — від отримання тригера до закриття кейсу.

`code`

    graph TD
    %% Початкова точка
    Start(( )) --> Receive[Отримання сповіщення про інцидент]
    
    Receive --> Analyze{Аналіз критичності}
    
    %% Гілки прийняття рішень
    Analyze -- Висока --> AutoIso[Автоматична ізоляція вузла]
    Analyze -- Низька --> Monitor[Посилений моніторинг логів]
    
    %% Точка злиття (Merge)
    AutoIso --> Collect[Збір форензик-даних та артефактів]
    Monitor --> Collect
    
    Collect --> Remediation[Спроба автоматичного виправлення]
    
    Remediation --> Result{Результат?}
    
    %% Друге рішення
    Result -- Успішно --> Restore[Відновлення доступу та зняття обмежень]
    Result -- Помилка --> Alert[Оповіщення експерта/SOC-аналітика]
    
    Restore --> Archive[Архівавання та закриття кейсу]
    Alert --> Archive
    
    %% Кінцева точка
    Archive --> EndNode((( )))

    %% Стилізація для кращого вигляду
    style Start fill:#000,stroke:#333
    style EndNode fill:#000,stroke:#333
    style Analyze fill:#fff2cc,stroke:#d6b656
    style Result fill:#fff2cc,stroke:#d6b656
    
`code`

### Пояснення змісту та зв’язків
**Логічний зв'язок:**

* **Use Case** задає межі системи: ми бачимо, що адміністратор може ізолювати вузли та збирати дані.

* **Sequence** деталізує один з цих кейсів ("Ізоляція"), показуючи, як саме повідомлення передаються від UI до агента на хості.

* **Activity** показує загальний бізнес-процес: як система приймає рішення (вибір шляху залежно від критичності), що об'єднує декілька варіантів використання в єдиний робочий цикл.

**Використані нотації:**

* В Sequence використано autonumber для чіткості кроків та Note для опису внутрішніх станів.

* В Activity використано стани вибору (Decision points) для відображення логіки реагування.
